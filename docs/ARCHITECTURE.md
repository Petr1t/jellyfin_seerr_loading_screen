# Architecture

## Overview

`jellyfin_seerr_loading_screen` is split into two independent halves:

1. **`jslsd`** — a Python daemon that talks to your Arr stack and exposes a clean JSON API + poster images.
2. **`plugin`** — a C# Jellyfin plugin that consumes the daemon's API and renders virtual items inside Jellyfin's library view.

This split is intentional. The daemon is the hard, IO-bound part: polling Sonarr/Radarr APIs, normalising queue states, fetching TMDB/TVDB poster art, generating overlays. None of that needs to know about Jellyfin's plugin system. The plugin is a thin display adapter that polls the daemon and creates virtual `BaseItem`s.

## Why a side-car instead of doing it all in the C# plugin?

Three reasons:

1. **Iteration speed.** Python lets us prototype poster generation, queue normalisation, and ETA heuristics in hours. C# inside Jellyfin's plugin sandbox has a longer feedback loop.
2. **Reusability.** The daemon's `GET /api/coming-soon` is useful even without Jellyfin — you could feed it into a HUB75 LED display, a Telegram bot, a smart-home dashboard. People who don't run Jellyfin still get value.
3. **Clear failure isolation.** If the daemon crashes, Jellyfin keeps working. If Jellyfin restarts, the daemon doesn't lose its in-memory cache (state is reconstructed on next poll within 30s).

```

│   ├── __init__.py

│   ├── arr_client.py      # Sonarr/Radarr REST clients

│   ├── tmdb_art.py        # iTunes/TMDB fallback for poster art when Sonarr/Radarr lack it
│   ├── api.py             # FastAPI app
│   └── models.py          # Pydantic types
├── pyproject.toml
└── tests/
    ├── conftest.py
    ├── test_arr_client.py
    ├── test_poster.py
    └── test_api.py
```

## State model

`PendingItem` (`models.py`):

```python
class PendingItem(BaseModel):
    id: str                     # f"{source}-{queue_id}", e.g. "sonarr-12345"
    source: Literal["sonarr", "radarr"]
    type: Literal["movie", "tv"]
    title: str                  # canonical human title
    series_title: str | None    # for TV episodes
    season: int | None
    episode: int | None
    tmdb_id: int | None
    tvdb_id: int | None
    size_total_bytes: int
    size_left_bytes: int
    progress_percent: float     # 0..100
    eta_seconds: int | None
    download_client: str        # "qbittorrent", "deluge", ...
    status: Literal["queued","downloading","completed","failed","paused"]
    requested_by: str | None    # Jellyseerr username if available
    poster_url: str             # jslsd-internal URL to overlay PNG
    last_updated: datetime
```

Cache lifecycle:
- **Queued items** → kept in cache, polled every 30s.
- **Completed items** → kept for 5 minutes, then evicted (so users see the "just finished" state briefly).
- **Failed items** → kept for 1 hour, with `status=failed` so UI can show retry option.

## Why progress as a generated PNG instead of CSS overlay?

Two reasons:
1. **Native Jellyfin clients (iOS, tvOS, Android TV) don't run our JS overlay.** They fetch the item's primary image. If we bake the progress into the image, all clients see it for free.
2. **Caching simplicity.** The PNG is a function of `(item_id, progress_bucket)` where bucket is `progress_percent // 5` (steps of 5%). We regenerate at most 21 PNGs per item lifecycle — trivial cost.

The Web UI plugin can additionally do live JS-based animation if it wants, but the static PNG is the floor.

## Webhooks (planned)

Polling every 30s is fine for v0.1. v0.3 will optionally accept Sonarr/Radarr webhooks for push-updates:

```
Sonarr/Radarr ──webhook──▶ jslsd POST /webhook/{provider}
                                    │
                                    ▼
                             invalidate cache for item
                             trigger immediate refresh
```

This drops latency from up-to-30s to sub-second.

## Multi-user

For Pet's home setup (one user) this is irrelevant. For multi-user setups it matters: housemates shouldn't see each other's pending requests by default.

Plan:
- Jellyseerr's `/api/v1/request` endpoint exposes `requestedBy.jellyfinUserId`.
- `jslsd` enriches each `PendingItem` with `requested_by_jellyfin_id`.
- The plugin queries Jellyfin's current user and filters.
- A config option `show_all_users` lets admins override.

## Testing strategy

- **Unit tests** for `arr_client.py`, `posters.py`, `tmdb_art.py` — pure Python, no external services.
- **Contract tests** for the FastAPI endpoints using `httpx.AsyncClient` against a live `app`.
- **Mock fixtures** for Sonarr/Radarr responses (recorded via `vcrpy` against a real instance once, replayed deterministically).
- **No integration tests against real Sonarr/Radarr** in CI — those run locally on Pet's paradecentral box (`make smoke-test`).

## Performance targets

- Daemon memory footprint: <100MB (one Python process, in-memory cache for ~50 pending items max).
- Poll cycle latency: <2s end-to-end for a queue of 50 items.
- Poster generation: <500ms per item (PIL with cached poster art).
- API response time: <50ms p95 for `/api/coming-soon`, <100ms for poster PNG (served from disk cache).
